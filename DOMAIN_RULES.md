# DOMAIN_RULES

Этот файл является source of truth для критичной доменной логики `car-service-platform`.

- Last updated: `2026-03-14`
- Status: `initial project baseline`

## 1) Core Entities
Текущий baseline сущностей:
- `Customer`
- `Vehicle`
- `RepairOrder`
- `RepairWork`
- `RepairPart`
- `Supplier`
- `RepairPhoto`
- `CompletionAct`

Для каждой сущности фиксировать:
- цель
- обязательные поля
- владельца source of truth
- допустимые статусы

### Initial Entity Notes
- Один `Customer` может иметь несколько `Vehicle`.
- Один `Vehicle` принадлежит одному активному `Customer` в рамках первой версии.
- `Vehicle` должен по возможности опираться на управляемый справочник марки, модели и года.
- Удаление `Customer`, у которого уже есть `Vehicle`, должно блокироваться до явного удаления или переназначения автомобилей.
- Один `RepairOrder` относится к одному `Vehicle` и одному `Customer`.
- Один `RepairOrder` может содержать несколько `RepairWork`, `RepairPart` и `RepairPhoto`.
- Каждый `RepairOrder` должен иметь отдельный публичный `tracking code` для клиента.
- `CompletionAct` формируется на основании завершенного `RepairOrder`.

## 2) Lifecycle Rules
Базовый lifecycle ремонта для первой версии:
- `draft -> diagnosed -> in_progress -> completed -> delivered`

Правила:
- `draft`: запись создана, проблема зафиксирована, состав работ еще не финализирован.
- `diagnosed`: первичная диагностика завершена, состав работ и запчастей начал формироваться.
- `in_progress`: ремонт выполняется, добавляются фактические работы, фото и запчасти.
- `completed`: ремонт завершен, итоговые суммы зафиксированы, можно формировать акт.
- `delivered`: автомобиль выдан клиенту, запись считается закрытой для операционного потока.

Дополнительно:
- переходы назад из `completed` и `delivered` не должны происходить без явного административного действия
- `CompletionAct` может быть сформирован только для `completed` или `delivered`
- месячная история должна строиться по данным `RepairOrder` и связанных сущностей, а не из отдельных ручных отчетов

## 3) Pricing Rules
Базовые правила первой версии:
- итог ремонта = сумма `RepairWork.cost` + сумма `RepairPart.cost`
- суммы по работам и запчастям считаются на backend как source of truth
- для `RepairPart` нужно различать закупочную стоимость и цену продажи клиенту
- сумма по запчастям для клиента считается от цены продажи, а не от закупочной стоимости
- UI может показывать предварительный расчет, но не должен быть источником итоговой суммы
- если появятся скидки, налоги, fees и округления, они должны быть описаны здесь до реализации

## 4) Access Rules
До появления полноценной auth-модели использовать минимальный baseline ролей:
- `admin`: полный доступ
- `manager`: работа с клиентами, автомобилями, ремонтами, актами
- `staff`: обновление ремонта, фото, работ и запчастей без системных настроек
- `client`: доступ только к просмотру статуса ремонта и связанных материалов по `tracking code`, без отдельного кабинета

Критичные ограничения:
- закрытие ремонта и формирование акта должны быть ограничены ролями `admin` и `manager`
- прямое изменение итоговых сумм вне source-of-truth логики запрещено
- после статуса `delivered` запись не должна редактироваться без отдельного override-механизма
- клиент не должен попадать в staff surface и не должен иметь доступ к Django Admin
- публичный клиентский доступ не должен использовать последовательный ID ремонта как единственный идентификатор

## 5) Integration Rules
На старте проекта внешние интеграции не входят в обязательный scope.

Если интеграции будут добавлены:
- инициатор и источник истины должны быть описаны явно
- retry, timeout и обработка частичных сбоев должны фиксироваться здесь до реализации

## 6) Confirmed Product Rules From Source Document
- система должна поддерживать глобальный поиск по номеру машины и имени клиента
- номер автомобиля должен быть уникальным в реестре первой версии
- карточка автомобиля должна показывать историю работ по машине
- карточка клиента должна показывать список автомобилей и историю обращений
- внутри ремонта должны храниться фото до, во время и после ремонта
- фото должны поддерживать мобильную съемку, дозагрузку новых файлов и удаление ошибочных кадров
- внутри ремонта должны храниться работы, запчасти и поставщики
- для заказанных запчастей должны храниться дата заказа, закупочная цена, количество, сумма закупки, цена продажи клиенту и привязка к ремонту / автомобилю
- месячная история должна поддерживать агрегаты по суммам и количеству объектов
- акт выполненных работ должен содержать клиента, автомобиль, таблицу работ, таблицу запчастей и итоговую сумму

## 7) Open Decisions
- хранить ли `email` клиента в первой версии
- может ли один `RepairOrder` содержать несколько отдельных проблем как структурированные элементы
- нужен ли fallback в ручной ввод, если нужной машины нет в справочнике
- нужен ли учет оплат клиента
- нужен ли учет сотрудников, выполняющих конкретные работы
- нужен ли склад запчастей
- обязателен ли PDF-экспорт акта в первой версии
- нужен ли VIN-enrichment после MVP и какой провайдер будет использоваться
- нужны ли уведомления о ТО и запланированном ремонте

## 8) Review Policy
`domain-rules-reviewer` обязателен, если меняются:
- статусы и переходы
- расчеты и totals
- права доступа
- критичные доменные инварианты
